def get_top_articles(n, df=df):
    """
    INPUT:
    n - (int) the number of top articles to return
    df - (pandas dataframe) df as defined at the top of the notebook 
    
    OUTPUT:
    top_articles - (list) A list of the top 'n' article titles
    
    Description:
    This function should return the top 'n' article titles based on the number of interactions
    """
    
    # Group by article_id and count the number of interactions for each article
    article_interactions = df.groupby('article_id').size().sort_values(ascending=False)
    
    # Get the top n article ids
    top_article_ids = article_interactions.index[:n].tolist()
    
    # Get the corresponding article titles from df (not df_content)
    top_articles = df[df['article_id'].isin(top_article_ids)]['title'].unique().tolist()[:n]

    INPUT:

def find_similar_users(user_id, user_item=user_item):
    '''
    INPUT:
    user_id - (int) a user_id
    user_item - (pandas dataframe) matrix of users by articles: 
                1's when a user has interacted with an article, 0 otherwise
    
    OUTPUT:
    similar_users - (list) an ordered list where the closest users (largest dot product users)
                    are listed first
    
    Description:
    Computes the similarity of every pair of users based on the dot product
    Returns an ordered list of the users in descending order of similarity
    '''
    
    # Compute similarity of each user to the provided user using dot product
    user_vec = user_item.loc[user_id].values  # vector of the provided user
    similarity = user_item.dot(user_vec)      # dot product to get similarities
    
    # Sort by similarity (highest similarity first)
    similarity_sorted = similarity.sort_values(ascending=False)
    
    # Create list of user ids from index
    most_similar_users = similarity_sorted.index.tolist()
    
    # Remove the own user's id
    most_similar_users.remove(user_id)
    
    return most_similar_users  # return a list of the users in order from most to least similar

def get_top_sorted_users(user_id, df=df, user_item=user_item):
    '''
    INPUT:
    user_id - (int) a user_id
    df - (pandas dataframe) interaction dataframe
    user_item - (pandas dataframe) user-item interaction matrix
    
    OUTPUT:
    neighbors_df - (pandas dataframe) a dataframe with:
                    - neighbor_id: ID of similar users
                    - similarity: Similarity score (dot product)
                    - num_interactions: Total interactions of the neighbor
    
    Other Details:
    - Sort the `neighbors_df` by similarity (descending), then by num_interactions (descending).
    '''
    
    # Compute similarity of each user with the provided user
    user_vec = user_item.loc[user_id].values
    similarity = user_item.dot(user_vec)
    
    # Create a DataFrame for neighbors
    neighbors_df = pd.DataFrame({'neighbor_id': user_item.index, 'similarity': similarity})
    
    # Exclude the current user from the neighbors
    neighbors_df = neighbors_df[neighbors_df['neighbor_id'] != user_id]
    
    # Get the number of interactions each user has
    num_interactions = df.groupby('user_id')['article_id'].count()
    neighbors_df['num_interactions'] = neighbors_df['neighbor_id'].map(num_interactions)
    
    # Sort by similarity first, then by number of interactions
    neighbors_df = neighbors_df.sort_values(by=['similarity', 'num_interactions'], ascending=False)

    return neighbors_df

        new_recs = np.setdiff1d(neighbor_articles, user_seen_articles, assume_unique=True)

        # Ensure only valid articles are considered
        valid_recs = [str(article) for article in new_recs if str(article) in article_interactions.index]

        # Apply stable sorting: Sort by interaction count (descending) and then by article ID (as float)
        sorted_recs = sorted(valid_recs, key=lambda x: (-article_interactions[x], float(x)))

        # Add sorted recommendations
        recs.extend(sorted_recs)

        # Stop if we have enough recommendations
        if len(recs) >= m:
            break

    # Limit to top `m` recommendations
    recs = recs[:m]

    # Ensure article names are retrieved in the same order
    rec_names = [df[df['article_id'].astype(str) == str(article_id)]['title'].iloc[0] for article_id in recs]

What SVD Does:

  • U (Users × Latent Features): Captures relationships between users and hidden factors.
  • S (Diagonal Singular Values): Contains the importance (weights) of each hidden factor.
  • Vt (Latent Features × Articles): Captures relationships between hidden factors and articles.

Why This is Different from the Lesson:

  1. Binary Interactions: Unlike the lessons, where we had explicit ratings (e.g., 1-5 ratings from movie users), our user-item matrix contains only binary values (0s and 1s) representing whether a user interacted with an article or not.

  2. Sparse Matrix: The matrix is highly sparse because most users have interacted with only a few articles. In lessons, we usually dealt with denser matrices.

  3. Missing Values: In previous lessons, missing values (i.e., articles a user hasn't rated) were explicitly handled. Here, missing values are just zeros in our matrix.

def get_recommendations_svd(user_id, U, Sigma, VT, user_item_matrix, m=10):
    '''
    INPUT:
    user_id - (int) a user id
    U - (numpy array) U matrix from SVD
    Sigma - (numpy array) Sigma matrix from SVD
    VT - (numpy array) V-transpose matrix from SVD
    user_item_matrix - (pandas dataframe) matrix of users by articles
    m - (int) the number of recommendations you want for the user
    
    OUTPUT:
    recs - (list) a list of recommendations for the user by article id
    rec_names - (list) a list of recommendations for the user by article title
    '''
    user_row = user_item_matrix.index.get_loc(user_id)
    user_vector = np.dot(U[user_row, :], np.diag(Sigma))
    scores = np.dot(user_vector, VT)
    
    # Get the indices of the top m scores
    top_indices = np.argsort(scores)[::-1][:m]
    
    # Get the article ids corresponding to the top indices
    recs = user_item_matrix.columns[top_indices].tolist()
    
    return recs, get_article_names(recs)

from sklearn.metrics import precision_score, recall_score, f1_score

def evaluate_recommendations(y_true, y_pred, k=10):
    '''
    INPUT:
    y_true - (list) Actual articles the user interacted with
    y_pred - (list) Articles recommended by the system
    k - (int) Number of recommendations made

    OUTPUT:
    precision, recall, f1_score - Performance metrics
    '''
    # Convert to sets
    y_true_set = set(y_true)
    y_pred_set = set(y_pred[:k])  # Only consider top K recommendations

    # Calculate Precision & Recall
    true_positives = len(y_true_set & y_pred_set)  # Correctly recommended articles
    precision = true_positives / len(y_pred_set) if len(y_pred_set) > 0 else 0
    recall = true_positives / len(y_true_set) if len(y_true_set) > 0 else 0

    # Compute F1 Score
    if precision + recall == 0:
        f1 = 0
    else:
        f1 = 2 * (precision * recall) / (precision + recall)

    return precision, recall, f1

Response for Section 6: Comment on Results & Testing Recommendation Performance¶

Observations from the Results¶

From the results of the Singular Value Decomposition (SVD) approach in Part V, we observed:

  1. Accuracy Trends:

    • As the number of latent features increases, the training accuracy improves.
    • However, test accuracy plateaus beyond a certain number of latent features.
    • This suggests that using too many latent features may lead to overfitting, where the model memorizes training interactions but does not generalize well for new users.

  2. Cold Start Problem:

    • We were only able to make recommendations for 20 users in the test set.
    • The remaining 662 users could not be predicted for due to a lack of historical interactions in the training data.
    • This is a classic cold start issue in recommendation systems.

  3. User-Item Matrix Sparsity:

    • Since the dataset consists only of binary interactions (1 or 0), the matrix is highly sparse.
    • The SVD method performs well for users with multiple interactions but struggles for users with only a few or no historical interactions.

How to Test the Effectiveness of Recommendations¶

To determine if the recommendation engine provides better results than the current system, we can use the following evaluation methods:

1. A/B Testing (Online Evaluation)¶

  • Split users into two groups:
    • Control Group → Receives recommendations from the existing system.
    • Treatment Group → Receives recommendations from the new system (SVD-based recommendations).
  • Measure Key Metrics:
    • Click-Through Rate (CTR) → How many recommended articles are clicked?
    • Engagement Time → How long do users spend on recommended articles?
    • User Satisfaction Surveys → Ask users to rate the relevance of recommendations.

Advantage: Measures real user behavior in a live environment.
Disadvantage: Requires deployment, which can take time.

2. Precision, Recall, and F1-Score (Offline Evaluation)¶

Since we have actual user interactions, we can measure how well our recommendations match real preferences.

  • Precision: Percentage of recommended articles that were actually interacted with by users. [ Precision = \frac{Relevant\ Recommendations}{Total\ Recommendations} ]

  • Recall: Percentage of articles that a user interacted with that were successfully recommended. [ Recall = \frac{Relevant\ Recommendations}{Total\ Interacted\ Articles} ]

  • F1-Score: A balance between precision and recall. [ F1 = 2 \times \frac{Precision \times Recall}{Precision + Recall} ]

Advantage: Can be computed without requiring user deployment.
Disadvantage: Does not measure actual user engagement.

3. Normalized Discounted Cumulative Gain (NDCG)¶

  • NDCG measures how highly relevant recommendations appear at the top of the ranked list.
  • The higher the NDCG, the better the model at ranking articles that users find useful.

Advantage: Measures ranking quality, not just correctness.
Disadvantage: Requires a large dataset with ranking information.

Addressing Class Imbalance in Recommendations¶

The class imbalance issue occurs because most articles are not interacted with (class 0), while only a few are popular (class 1).
To fix this:

  • Data Resampling: Balance training data by oversampling minority interactions.
  • Synthetic Data Generation: Use SMOTE (Synthetic Minority Over-sampling Technique) to create new interactions.
  • Class Weighting: Assign higher weights to underrepresented interactions in model training.

Advantage: Reduces bias toward popular articles and improves recommendation diversity.

Final Thoughts¶

  1. SVD works well for active users but fails for new users (cold start problem).
  2. A/B testing is the best way to validate if our recommendations improve engagement.
  3. Metrics like Precision, Recall, and NDCG can evaluate offline performance.
  4. Handling class imbalance will make recommendations more diverse and less biased.
